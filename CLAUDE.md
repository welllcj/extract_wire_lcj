# CLAUDE.md

## 项目目标
这是一个点云导线/导管提取项目。

当前任务是优化 `test.py` 中的以下函数，使其在复杂点云场景下更稳定、更高效地完成导线和导管提取：

- `extract_wire_curve_mode`
- `process_representative_points_and_neighbors`
- `estimate_structure_radius`

## 允许修改的范围
只允许修改 `test.py` 中的以下函数：

- `extract_wire_curve_mode`
- `process_representative_points_and_neighbors`
- `estimate_structure_radius`

如果确实绝对必要，可以在这几个函数内部新增局部嵌套辅助逻辑，但不要修改其他函数定义，不要修改函数接口。

## 禁止修改的范围
除上述允许修改的函数外，其他文件和其他函数都不要修改，包括但不限于：

- `newton.py`
- UI 相关代码
- 文件读写逻辑
- `extract_wire_point_mode`
- 其他未明确允许修改的内容

## 核心任务
用户会在目标导线或导管上手动点击若干种子点。

需要基于这些种子点，尽可能鲁棒地完成以下任务：

1. 正确估计导线或导管半径  
2. 正确提取整条目标导线或导管上的点  
3. 尽量不要误提取附近其他设备、构件或杂散点  
4. 对弯曲导线/导管保持稳定跟踪  
5. 对断断续续、稀疏、不完整的点云保持一定的连续提取能力  
6. 在点云规模较大时，尽量使用高效算法，减少不必要的全局重复计算  

## 成功标准
优化后的代码应尽量满足以下标准：

### 半径估计
- 能够根据用户提供的种子点估计目标导线或导管的半径
- 估计值与真实值的相对误差尽量不超过 30%

### 提取正确性
- 能够正确提取目标导线或导管上的点
- 尽量不要提取到其他设备上的点
- 提取结果应尽量沿着真实导线/导管连续延伸
- 不应出现明显不合理的突然剧烈弯折或跳跃

### 对低质量点云的适应性
- 当导线或导管点云存在局部稀疏、断裂、缺失时，不要轻易突然停止
- 应尽量在合理范围内继续沿正确方向跟踪
- 但不能为了跨越断裂而明显误吸附到别的结构上

### 效率
- 优先使用局部搜索、局部更新、高效数据结构（如 KDTree）
- 避免不必要的全局重拟合、重复邻域扫描和高复杂度重复计算
- 不强求不现实的理论复杂度指标，但应尽量将主要流程设计为工程上高效可行的局部算法
- 在保证正确性的前提下，尽量使整体复杂度接近局部邻域搜索主导的高效实现

## 推荐优化方向
优先考虑以下方向，而不是继续依赖单一全局高阶曲线拟合：

- 局部滑动窗口
- 局部低阶曲线拟合
- 局部切向估计
- 双向扩展
- 距离约束 + 方向连续性约束
- 面向中空导管的局部中心轴/局部管状几何判别
- 对短距离断裂的有限桥接机制
- 尽量减少全局重复拟合带来的时间开销

## 默认失败处理
如果当前一轮修改失败，或者效果明显变差，则不要保留坏方案。

处理规则：
1. 回退当前失败思路
2. 从最近一个稳定版本重新开始
3. 换一个更小、更稳的方向继续优化

不要在明显失败的基础上继续堆补丁。

## 默认提交规则
- 每隔 3 分钟检查一次是否形成新的稳定版本
- 只有在当前版本可运行、逻辑没有明显变差、结果不差于上一个稳定版本时，才允许提交
- 不要为了凑时间提交坏状态
- 提交信息要清晰描述本轮改动目标

## 默认沟通规则
默认不要向用户提问，也不要频繁请求确认。

除非触发以下硬例外，否则请直接继续执行：
1. 只修改允许范围内的函数无法完成任务
2. 必须修改其他文件或函数接口才能继续
3. 仓库或分支状态不安全
4. 出现会明显破坏项目的高风险操作

除此之外，请默认已获得执行授权，直接进行：
- 分析
- 修改
- 验证
- 比较
- 回退
- 继续优化

## 工作方式
请采用闭环工作流：

1. 先分析当前实现的问题
2. 提出一个明确的小步优化方案
3. 修改代码
4. 验证结果
5. 若效果更好则保留，否则回退
6. 持续迭代

不要一开始就做不可控的大重写，优先采用小步、可验证、可回退的工程化优化方式。
